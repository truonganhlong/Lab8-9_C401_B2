## 2. Tôi đã ra một quyết định kỹ thuật gì? (150–200 từ)

**Quyết định:** Tích hợp trực tiếp **Jina Embedding v5** vào Retrieval Worker và ép cấu hình `PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION=python`.

**Lý do:** 
Tôi chọn Jina v5 không chỉ vì nó mạnh về đa ngôn ngữ, mà vì nó hỗ trợ **Task-specific adapters**. Trong dự án này, tôi sử dụng adapter `retrieval.query` để định hướng không gian vector phù hợp với các câu hỏi ngắn đặc thù của Helpdesk. 

Ngoài ra, một quyết định "xương máu" là tôi phải ép cấu hình môi trường Protobuf trong code (`line 25, retrieval.py`). Trên môi trường Windows của nhóm, việc cài đặt `chromadb` thường xuyên xung đột với các thư viện local embedding gây lỗi *Protobuf version mismatch*. Việc dùng Jina API kết hợp với cấu hình ép kiểu Python implementation giúp hệ thống chạy ổn định 100% trên mọi máy của thành viên mà không cần cài đặt lại môi trường phức tạp.

**Trade-off đã chấp nhận:**
Vì gọi API quốc tế (Jina), tôi phải chấp nhận **Network Latency Overhead**. Mỗi lần retrieve mất ~1.5s - 2s. Để tránh việc pipeline bị treo vô hạn (causing graph failure), tôi đã phải thiết kế một "Graceful Fallback": Nếu API timeout sau 20s, worker sẽ trả về mảng rỗng (abstain) thay vì throw exception, giúp Synthesis vẫn có thể trả lời "Tôi không biết" thay vì làm sập toàn bộ hệ thống.

**Bằng chứng từ trace/code:**
Trong `retrieval.py` (Line 53-60):
```python
data = {
    "model": "jina-embeddings-v5-text-small",
    "input": [text],
    "task": "retrieval.query",
    "dimensions": 1024,
}
resp = requests.post(url, headers=headers, json=data, timeout=20)
```
Kết quả thực tế từ trace `run_20260414_164755.json` cho câu hỏi về SLA:
- **Workers called**: `["retrieval_worker", "synthesis_worker"]` -> Worker của tôi đã chạy chính xác.
- **Score**: `0.5726` cho file `sla_p1_2026.txt`. 
- **Latency**: Tổng graph mất `6589ms`, trong đó phần embedding Jina chiếm khoảng 1.8s, chứng tỏ tradeoff về thời gian là xứng đáng để đổi lấy độ chính xác cao (score > 0.5).

## 3. Tôi đã sửa một lỗi gì? (150–200 từ)

**Lỗi:** **ChromaDB CWD Path Mismatch** (Lỗi lạc trôi folder database).

**Symptom (pipeline làm gì sai?):**
Khi tôi chạy test độc lập `python workers/retrieval.py`, worker báo `Retrieved: 3 chunks` (chuẩn). Nhưng khi chạy tích hợp qua `python graph.py` từ thư mục gốc, hệ thống lại báo `Retrieved: 0 chunks`. Synthesis liên tục trả lời "Không đủ thông tin" một cách vô lý.

**Root cause (lỗi nằm ở đâu — indexing, routing, contract, worker logic?):**
Lỗi do dùng đường dẫn tương đối `./chroma_db`. Khi chạy script ở các tầng thư mục khác nhau, "điểm gốc" của đường dẫn tương đối bị thay đổi. 
- Chạy từ `workers/`: nó tìm `./workers/chroma_db` (không tồn tại -> tự tạo mới rỗng).
- Chạy từ root: nó tìm `./chroma_db` (có dữ liệu).
Điều này vi phạm tính nhất quán của Worker Contract.

**Cách sửa:**
Tôi đã sửa lại hàm `_get_collection()` để tính toán đường dẫn tuyệt đối dựa trên vị trí của chính file `retrieval.py`. Tôi dùng bộ đôi `os.path.dirname` và `os.path.abspath(__file__)` để "ghim" đường dẫn vào thư mục gốc của project bất kể Script được gọi từ đâu.

**Bằng chứng trước/sau:**
Code sửa đổi (Line 113-114):
```python
base_dir = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
chroma_path = os.path.join(base_dir, "chroma_db")
```
Sau khi sửa, trace ID `run_20260414_164755.json` (chạy từ root dự án) đã hiển thị đúng kết quả gọi worker:
- **System Log**: `[retrieval_worker] retrieved 3 chunks from ['sla_p1_2026.txt', 'policy_refund_v4.txt', 'access_control_sop.txt']`.
Nếu không dùng đường dẫn tuyệt đối, worker này khi gọi từ Graph (root) sẽ cố tìm database trong thư mục rỗng và trả về 0 chunks, khiến Synthesis không thể trả lời được các câu hỏi về SLA.

## 4. Tôi tự đánh giá đóng góp của mình (100–150 từ)

**Tôi làm tốt nhất ở điểm nào?**
Tôi tự tin nhất ở khả năng "giải quyết vấn đề" (troubleshooting). Tôi không chỉ dừng lại ở việc viết code theo template mà luôn thực hiện các bước debug sâu để hiểu tại sao hệ thống hoạt động không như ý, từ đó đưa ra các bản vá về đường dẫn tuyệt đối hay tăng timeout cho API giúp hệ thống chạy ổn định hơn nhiều.

**Tôi làm chưa tốt hoặc còn yếu ở điểm nào?**
Do tập trung quá nhiều vào logic của từng worker đơn lẻ, tôi chưa dành đủ thời gian để tối ưu hóa Supervisor routing. Supervisor hiện tại vẫn thuần rule-based, đôi khi còn cứng nhắc khi gặp các câu hỏi có từ khóa giao thoa.

**Nhóm phụ thuộc vào tôi ở đâu?**
Nếu phần Worker của tôi chưa xong hoặc bị lỗi contract, Supervisor sẽ không có ai để "sai việc", và Synthesis sẽ không có context để trả lời. Toàn bộ engine thực thi của Helpdesk nằm ở 3 file worker chính mà tôi phụ trách.

**Phần tôi phụ thuộc vào thành viên khác:**
Tôi phụ thuộc vào TECH LEAD để định nghĩa Metadata cho các tệp tài liệu và cung cấp bộ API Keys ổn định. Nếu Tech Lead không thông báo thay đổi logic state (ví dụ đổi tên biến `task` thành `query`), code của tôi sẽ bị lỗi `KeyError` ngay lập tức.
